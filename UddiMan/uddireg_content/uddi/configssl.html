Configuring WebSphere to use HTTPS and SSL

To support the use of secure access with the IBM WebSphere UDDI Registry, you need to configure WebSphere to use HTTPS and SSL. This section describes a simple way in which you can do this for WebSphere Application Server AE, using the WebSphere sample certificates. Please note that this configuration should never be used in a production environment. A more comprehensive way of setting up SSL is to configure SSL through the web server, as described in section 6.6.18.1a.7 Configuring SSL in WebSphere Application Server in the WebSphere Application Server Advanced Edition InfoCenter.

Note: The following instructions are for WebSphere Application Server AE; if you are using AEd then refer to the AEd documentation.

To configure WebSphere to use HTTPS and SSL, complete the following steps:

  1. Set aliases of your Virtual Host:
    1. On the WAS AE Administrative Console, in the tree on the left-hand side click on Virtual Hosts...-> Default Host (or whichever host you have set up the UDDI Registry to use if you are not using the default one.)
    2. On the "General" tab, add a new host alias, called "*:9443". (9443 is the default WAS SSL port.) This is in addition to the aliases "*:9080" and "*:80" which should already be setup. (Make sure that there are colons and not full stops between the * and the number.)
    3. Click "Apply".
  2. Add a new HTTP transport
    1. In the tree, go to Nodes... -> machine name... -> Application Servers... -> Default Server (or whichever server you have set up to use if not the default one.)
    2. Go to the "Service" tab, select "Web Container Service", and press "Edit properties".
    3. Go to the "Transport" tab. There should be at least one transport already defined, with host=* and port=9080.
    4. Press "Add" to add a new transport. On the "General" tab, enter "*" as the host and "9443" as the port. Check the "Enable SSL" and "Use Global SSL Default Configuration" checkboxes. Press "OK".
    5. Press "OK" again.
    6. Press "Apply".
  3. Check Default SSL Configuration - needed for Samples

    The SSL configuration uses key files and trust files. If you haven't changed these then the initial default values are adequate and will work with the samples. However, if you have changed the trust and key values, then you will either need to edit the SOAP sample code, or check that your SSL configuration is set up appropriately as follows:

    1. Go to the Console... -> Security Center menu.
    2. Go to the "General tab and click the ellipsis "..." button next to "Default SSL Configuration".
    3. Go to the "General" tab. The settings should be:
      • Key file name = ${WAS_HOME}/etc/DummyServerKeyFile.jks (the password for this file is "WebAS")
      • Key file format = JKS
      • Trust file name = ${WAS_HOME}/etc/DummyServerTrustFile.jks (the password for this file is "WebAS")
      • Trust file format = JKS

        You don't need to enable client authentication and the security level can be left as "high".

  4. For SOAP samples to work, you need to ensure that the Client JDK is either the one shipped with IBM WebSphere Application Server or a later IBM JDK.:

    • For Windows - ensure that C:\WebSphere\AppServer\java\bin is present in the PATH statement before and other JDK's
    • For Linux - ensure that /opt/WebSphere/AppServer/java/bin is present in the PATH statement before and other JDK's

    NOTE: You must use the IBM WebSphere supplied JDK or a later level of the IBM JDK.

    For Windows, the default system path can be set via Control Panel ...-> Settings ...-> System ...-> Advanced Properties ...-> Environment Variables
    Alternatively, this can be accomplished just for the shell where you plan to run the samples by modifying the path within the shell:

    Windows
    set path=c:\WebSphere\AppServer\java\bin;%path%
    Linux
    export PATH=/opt/WebSphere\AppServer/java/bin:$PATH
  5. Stop and restart the Default Server.
