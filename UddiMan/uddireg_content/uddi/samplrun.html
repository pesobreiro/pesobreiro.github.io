Configuring, compiling, and running the samples

This topic describes how to configure, compile, and run the samples provided with the IBM WebSphere UDDI Registry. It also provides examples of the output created by the samples.

Configuring the samples

Each sample has seven fields which need to be configured before compilation. These are:

private static final String host
= "hostname"; The host must be the name of the registry host machine, without the prefix "http://", e.g. "localhost" or "aspidistra.hursley.ibm.com"
private static boolean WASSecurityOn
= false; If WAS Security is turned on, this should be set to 'true'. Otherwise it should be left as 'false'
private static String publishUserName
= "db2admin";
private static String publishPassword
= "db2admin"; Username and password of an authenticated WAS user
private static final String trustStoreDirPath =
"c:\\websphere\\appserver\\etc\\"; On Linux this value should be set, by default, to "/opt/WebSphere/AppServer/etc"
private static final String trustStoreFilename =
"DummyClientTrustFile.jks"; trustStoreDirPath + trustStoreFilename must be fully qualified path to WAS's default dummy client trust file. If WAS is running remotely, the easiest way to run the samples is to get a copy of this file and place it in the same directory as the samples
private static final String trustStorePassword =
"WebAS"; This must be the password of the client trust file above

SSL

If WebSphere Application Server security is turned on, the samples must use SSL to access the publish API. If security is turned off, the samples do not use SSL at all.

The last three fields are to do with SSL certification. Setting the client trust store file to be the WebSphere Application Server default dummy client trust file allows the sample to trust the dummy signed certificate that WebSphere Application Server is using to serve SSL requests. If WebSphere Application Server has not been installed into the default directory, or it has been configured to use an alternative certificate, these fields will need to be configured. A discussion of SSL/Java certificates is beyond the scope of this sample - if running with alternative certificates installed on the server, the user must discuss with their server administrator the correct method of obtaining the required public certificates.

Setting the classpath

The java classpath needed to run the SOAP sample does not need any extra classes, apart from the standard java class libraries, added when using the WebSphere JDK. Using the WAS JDK, ensure that the file ibmjsse.jar is on the classpath. This jar file is required for SSL.

For the UDDI4J samples, the java classpath needed to run the sample must contain, apart from the standard java class libraries, xerces.jar, soap.jar, and the uddi4j.jar supplied with the IBM WebSphere UDDI Registry. It is essential to use this uddi4j.jar file rather than the one that can be downloaded from the UDDI4J website, as this version has been configured for use with WebSphere Application Server Security.

The locations for these files, by default, are shown below:

Windows
C:\WebSphere\AppServer\java\jre\lib\rt.jar
C:\WebSphere\AppServer\java\jre\lib\ext\ibmjsse.jar
C:\WebSphere\AppServer\lib\xerces.jar
C:\WebSphere\AppServer\lib\soap.jar
C:\WebSphere\UDDIReg\uddi4j\uddi4j.jar
Linux
/opt/WebSphere/AppServer\java\jre\lib\rt.jar
/opt/WebSphere/AppServer/java/jre/lib/ext/ibmjsse.jar
/opt/WebSphere/AppServer/lib/xerces.jar
/opt/WebSphere/AppServer/lib/soap.jar
/opt/WebSphere/UDDIReg/uddi4j/iddi4j.jar

Compiling the samples

To compile the samples, make sure you are using the WebSphere Application Server JDK. This can be done by setting the path before any other path to a JDK, as follows:

Windows
C:\WebSphere\AppServer\java\bin
Linux
/opt/WebSphere/AppServer/java/bin

Ensure the classpath is set up correctly, change to the appropriate directory, and, irrespective of operating system, type:

Windows example
cd C:\WebSphere\UDDIReg\samples\soap
javac SOAPSample1*.java

for the SOAP samples, or, for the UDDI4J samples:

Linux example
cd /opt/WebSphere/UDDIReg/Samples/soap
javac UDDI4JSample1*.java

Running the samples

Ensure you are using the WebSphere Application Server JDK , as above. Ensure the XML files containing the UDDI API messages are in the same directory as the java class files and type

java SOAPSample1a
java SOAPSample1b

and so on. The samples will send output to the console.

For the UDDI4J samples, again ensure you are using the WebSphere Application Server JDK, as above, and type:

java UDDI4JSample1a <WASUsername> <WASPassword>
java UDDI4JSample1b <WASUsername> <WASPassword>

and so on, where <WASUsername> and <WASPassword> are a valid WebSphere Application Server username and password pair. Again, the samples will send output to the console.
