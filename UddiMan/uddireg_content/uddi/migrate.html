Migrating to Version 1.1.1 of the IBM WebSphere UDDI Registry

Behavioural changes

  1. There have been a few minor changes to the EJB interface which mean that you may need to rewrite your EJB client applications:
    1. Remove any declarations that involve any of the classes in the com.ibm.uddi.exceptions package. There are only 5 Exceptions:
      • DuplicateLanguageCodeException
      • EmptyNameException
      • PersistenceException
      • TooManyConnectionsException
      • TooManyNamesException

      This includes trying to do a throw/catch or throws for any of these Exceptions, and also trying to import the package, or trying to instantiate/subclass any of the Exceptions.

    2. In the BusinessEntity class, a user cannot set the name of the business by doing an exampleBusinessEntity.setName(String aString);

      The user must now do an exampleBusinessEntity.setNames(com.ibm.uddi.datatypes.NameList aList); (please note that it is setNames and not setName as before)

  2. Some changes have occured with the behaviour of the UDDI User Console with regard to SSL and authentication:
    1. With authentication disabled, it is now possible to inquire and publish via HTTP on port 9080, or HTTPS on port 9443 (if configured).
    2. With authentication enabled, it is possible to inquire over HTTP without authenticating. Publish actions require access via HTTPS on port 9443 and you will be asked for authentication.
  3. tModels are now called technical models

Migration procedure

If you have previously installed version 1.1 of the IBM WebSphere UDDI Registry, then you should take the following steps in order to migrate to version 1.1.1.

UDDI Version 1.1.1 requires a new level of WebSphere Application Server, Version 4.0.3. It is recommended that you upgrade in two steps:

  • Upgrade to WebSphere Application Server 4.0.3 and then confirm that UDDI will still run as expected.
  • Upgrade to UDDI Version 1.1.1
  1. Upgrade to WebSphere Application Server 4.0.3.
  2. If you have made any changes to the configuration properties in the file uddi.properties, which is located in the properties subdirectory of your WebSphere AppServer install tree, then you should make a copy of this file (or make a note of all the changes), so that you will be able to reapply the changes to the file after you have installed the new release. The uddi.properties file will be replaced by the installation procedure.
  3. Uninstall the IBM WebSphere UDDI Registry using Add/Remove Programs on Windows platforms, or rpm -e IBMWebSphere-UDDI on Linux. This will remove the application, but will preserve the UDDI Registry database.
  4. Please note that you will be prompted during the installation procedure as to whether you want to replace the UDDI Registry database. If you have any data in the UDDI Registry which you wish to preserve, then you should reply "No" when prompted "Do you want to create a new UDDI Registry database?" and then confirm this choice.
  5. You can now follow the instructions on "Installation on Windows Platforms" or "Installation on Linux Platforms".
  6. After completing the install procedure, you should edit the uddi.properties file to reflect any changes that you require to the configuration properties. You should not replace this file with your previous copy from version 1.1 of the IBM WebSphere UDDI Registry as its format has changed.
